🚧  Public Beta

Long-running tasks, without the infrastructure

Render Workflows lets you break complex jobs into tasks that each run in their own instance — with queuing, retries, and orchestration handled for you. Define tasks as functions. Trigger them from anywhere. Chain them together. Pay per second.

The problem Workflows solves

When your app needs to run work that's long-running, multi-step, or compute-intensive, you typically end up building a lot of infrastructure around the actual logic. Workflows handles that infrastructure so you can focus on your code.

🔄

Durability across failures

A transient API timeout or a crashed process shouldn't lose your progress. Workflows provides durable execution — each task runs in its own isolated instance, and if it fails, only that task is affected. Parent tasks and completed siblings continue uninterrupted. Your pipeline doesn't restart from scratch because one step hit a blip.

🔧

Automatic retries with backoff

When a task throws an error, Workflows automatically retries it with configurable exponential backoff — no retry logic to write, no dead-letter queues to manage. Set maxRetries, waitDurationMs, and backoffScaling per task. A flaky third-party API call retries that step, not the whole job.

⚙️

No infrastructure to manage

No message brokers, no worker pools, no queue consumers. Render provisions a dedicated instance per run, executes your task, and deprovisions when it's done. You don't manage Redis, RabbitMQ, or SQS — just write functions and deploy.

💰

Per-second billing, zero idle cost

You pay only for the seconds your tasks actually run. No always-on worker instances sitting idle between jobs. Each task gets its own compute tier — a lightweight orchestrator on starter, a heavy transform on pro — so you never overpay for resources you don't need.

Your code stays focused on what to compute. Render handles how it gets orchestrated, retried, and provisioned.

What people build with Workflows

Workflows is designed for jobs where individual steps are long-running, benefit from independent compute, and need orchestration between them. Here are the patterns we see most:

🤖

AI agents with tool calling

An orchestrator task calls an LLM, interprets the response, and chains specialized tool tasks — web search, code execution, data retrieval — each in its own instance with its own compute specs and retry policy. The agent loops until it reaches a final answer. Each tool call is isolated so a transient API failure retries that step, not the whole conversation.

🔄

ETL and data pipelines

Extract records from an API, transform each batch in parallel across multiple instances, then load the results into Postgres. Give the extract step a starter instance and the compute-heavy transform step a pro instance. If a batch fails, the retry config re-runs just that batch — not the entire pipeline.

🖼️

Media and document processing

Fan out image resizing, video transcoding, or PDF extraction across dozens of parallel instances. A parent task splits a batch of URLs, chains a processImage task per URL using Promise.all, then collects the results. Each child gets its own 2-CPU instance and up to 24 hours to complete.

🌐

Batch API enrichment

Enrich thousands of CRM records by calling third-party APIs — geocoding, company data, email verification. Each API call runs as an isolated task with per-call retries and rate-limit-friendly exponential backoff. The parent task coordinates the fan-out and aggregates enriched records back into your database.

Get started in 5 minutes

Go from zero to a running workflow with the Render CLI. You'll scaffold a project, deploy it, and trigger your first run.

1. Install the Render CLI

brew update
brew install render
curl -fsSL https://raw.githubusercontent.com/render-oss/cli/refs/heads/main/bin/install.sh | sh

2. Scaffold your workflow

# Interactive: choose language, template, directory
render workflows init

# Or non-interactive (great for CI / coding agents):
render workflows init \
  --confirm \
  --language ts \
  --template hello-world \
  --dir my-workflow \
  --install-deps \
  --git

This generates a project with your first task definition, dependencies installed, and a Git repo initialized. The Hello World template gives you a minimal task you can modify immediately.

3. Push and deploy

# Commit and push to GitHub / GitLab / Bitbucket
cd my-workflow
git add . && git commit -m "Initial commit"
git remote add origin <your-repo-url>
git push -u origin main

Then in the Render Dashboard, click New → Workflow, link your repo, and click Deploy Workflow. Render builds your project and automatically registers all tasks it finds.

4. Trigger your first run

From the Dashboard, click into your task and hit Start Task. Provide input as a JSON array (e.g. [5]) and watch the run execute. Or trigger from code:

import { Render } from '@renderinc/sdk'

const render = new Render()
const result = await render.workflows.runTask(
  'my-workflow/calculateSquare', [5]
)
console.log(result.results) // 25
from render_sdk import Render

render = Render()
result = render.workflows.run_task(
    "my-workflow/calculate_square", [5]
)
print(result.results)  # 25
Next steps: Edit the generated task file to add your own logic, configure retry/timeout/instance type, and chain tasks together. See the full tutorial for a detailed walkthrough with screenshots.

How it works

The model is simple: you write functions, Render turns them into independently executable tasks that each run in their own provisioned instance.

Define → Deploy → Trigger → Execute (chain optionally) → Result

Key properties of the execution model:

Instance-per-run

Every run gets a dedicated instance. No noisy neighbors. Choose from 0.5 CPU to 16 CPU per task.

Sub-second spin-up

Render provisions each run's instance in under a second. No cold starts to manage.

Up to 24 hours

Each run can execute for up to 24 hours. Configurable timeout per task from 30s to 24h.

Auto-deprovisioned

When a run completes (or fails after retries), its instance is cleaned up. No idle cost.

Architecture

Runs execute inside the Render platform and can reach your Postgres databases, Redis caches, and other services over the private network. They can also call external APIs over the public internet. Runs cannot receive incoming connections.

Defining tasks

A task is a function you register with the Render SDK. You declare its name, compute size, timeout, and retry behavior alongside the function itself. Arguments and return values must be JSON-serializable.

import { task } from '@renderinc/sdk/workflows'

// Each task declaration = one independently runnable unit
const enrichCompany = task(
  {
    name: 'enrichCompany',
    plan: 'starter',           // 0.5 CPU, 512 MB — lightweight API calls
    timeoutSeconds: 300,       // 5 min max
    retry: {
      maxRetries: 4,           // Up to 5 total attempts
      waitDurationMs: 2000,    // 2s initial backoff
      backoffScaling: 1.5,     // 2s → 3s → 4.5s → 6.75s
    },
  },
  async function enrichCompany(domain: string): Promise<CompanyData> {
    const res = await fetch(`https://api.clearbit.com/v2/companies/find?domain=${domain}`)
    if (!res.ok) throw new Error(`Clearbit ${res.status}`)  // triggers retry
    return await res.json()
  }
)
from render_sdk import Workflows

app = Workflows()

@app.task(
    plan="starter",              # 0.5 CPU, 512 MB
    timeout_seconds=300,        # 5 min max
    retry={
        "max_retries": 4,
        "wait_duration_ms": 2000,
        "backoff_scaling": 1.5,
    },
)
async def enrich_company(domain: str) -> dict:
    res = await httpx.get(
        f"https://api.clearbit.com/v2/companies/find?domain={domain}"
    )
    res.raise_for_status()  # triggers retry on failure
    return res.json()
Multi-file organization: Spread tasks across files. In TypeScript, import them from your entry point. In Python, compose apps with Workflows.from_workflows().

Chaining and fan-out

A running task can call other task functions. Each call triggers a new run in its own instance. The caller gets a Promise (TS) or awaitable (Python) for the result. Use Promise.all or asyncio.gather to fan out work in parallel.

Each enrichCompany call spins up a separate instance. Promise.all awaits all in parallel.

Serial by default. Without Promise.all or asyncio.gather, chained runs execute one after another. Always wrap parallel work explicitly.

Triggering runs

Trigger runs from any TypeScript or Python app using the Render SDK, from any language via the REST API, or manually from the Dashboard and CLI.

import { Render } from '@renderinc/sdk'

const render = new Render()  // reads RENDER_API_KEY from env

// Fire-and-forget: kicks off the run, returns immediately
const started = await render.workflows.startTask(
  'my-workflow/enrichCompany',
  ['stripe.com']
)

// Retrieve the result when ready
const finished = await started.get()
console.log(finished.results)  // CompanyData JSON

// Or: trigger + wait in one call
const result = await render.workflows.runTask(
  'my-workflow/enrichCompany',
  ['stripe.com']
)
from render_sdk import RenderAsync

render = RenderAsync()

started = await render.workflows.start_task(
    "my-workflow/enrich_company", ["stripe.com"]
)
finished = await started
print(finished.results)
# Works from any language
curl -X POST https://api.render.com/v1/workflows/tasks/run \
  -H "Authorization: Bearer $RENDER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"taskIdentifier":"my-workflow/enrichCompany","inputData":["stripe.com"]}'

The task identifier is always {workflow-slug}/{task-name} — find it in the Dashboard on each task's page.

const run = await render.workflows.getTaskRun('trn-xyz789')
console.log(run.status, run.results)

// Cancel a running task
await render.workflows.cancelTaskRun('trn-xyz789')

Stream completion events

Subscribe to run completions over Server-Sent Events (SSE) with taskRunEvents(). Useful for monitoring dashboards or triggering downstream actions.

const controller = new AbortController()

  console.log(`Run ${event.id} → ${event.status}`)

Pricing

You pay for compute time (per-second, per instance type) and optionally for added concurrency. No charges when runs aren't executing.

Example: A standard task that runs for 30 seconds costs $0.20 × (30/3600) = $0.0017. Running 10,000 of those per month costs ~$17.

Guide

Processing video & large files

Fan out media processing across parallel instances — resize images, transcode video, or extract text from PDFs — with per-task compute sizing and automatic retries.

What you'll build

A workflow that accepts a list of file URLs, fans out processing to parallel instances, and returns the results. The pattern works for any kind of file — we'll use video transcoding as the example.

Parent fans out to N parallel instances, each processing one file. Results aggregated and uploaded.

Task 1: Transcode a single video

Runs on a pro instance (2 CPU, 4 GB). Downloads the source, transcodes with FFmpeg, uploads the result.

import { task } from '@renderinc/sdk/workflows'
import { execSync } from 'child_process'
import { uploadToS3, downloadFile } from './storage'

export const transcodeVideo = task(
  {
    name: 'transcodeVideo',
    plan: 'pro',              // 2 CPU, 4 GB — enough for 1080p
    timeoutSeconds: 3600,
    retry: { maxRetries: 2, waitDurationMs: 5000, backoffScaling: 2 },
  },
  async function transcodeVideo(sourceUrl: string, outputFormat = 'mp4') {
    const inputPath = await downloadFile(sourceUrl, '/tmp/input')
    const outputPath = `/tmp/output.${outputFormat}`
    const start = Date.now()
    execSync(`ffmpeg -i ${inputPath} -c:v libx264 -preset fast -crf 23 ${outputPath}`)
    const outputUrl = await uploadToS3(outputPath)
    return { outputUrl, durationMs: Date.now() - start }
  }
)
import subprocess, time
from render_sdk import Workflows
from storage import download_file, upload_to_s3

app = Workflows()

@app.task(plan="pro", timeout_seconds=3600,
    retry={"max_retries": 2, "wait_duration_ms": 5000, "backoff_scaling": 2})
async def transcode_video(source_url: str, output_format: str = "mp4") -> dict:
    input_path = await download_file(source_url, "/tmp/input")
    output_path = f"/tmp/output.{output_format}"
    start = time.time()
    subprocess.run(["ffmpeg", "-i", input_path, "-c:v", "libx264",
        "-preset", "fast", "-crf", "23", output_path], check=True)
    url = await upload_to_s3(output_path)
    return {"output_url": url, "duration_ms": int((time.time() - start) * 1000)}

Task 2: Orchestrator — fan out a batch

export const processUpload = task(
  { name: 'processUpload', plan: 'starter', timeoutSeconds: 7200 },
  async function processUpload(videoUrls: string[], format = 'mp4') {
    const results = await Promise.all(
      videoUrls.map(url => transcodeVideo(url, format))
    )
    return results
  }
)
@app.task(plan="starter", timeout_seconds=7200)
async def process_upload(video_urls: list[str], format: str = "mp4") -> list:
    results = await asyncio.gather(
        *[transcode_video(url, format) for url in video_urls]
    )
    return list(results)
Cost insight: The orchestrator runs on a $0.05/hr starter instance while the heavy work runs on $0.40/hr pro instances — you only pay for the compute you actually need at each stage.

Adapting the pattern

Image resizing: Replace FFmpeg with Sharp or Pillow. Use standard instances. PDF extraction: Use pdftotext or Tika. Increase timeoutSeconds for scanned docs. Audio transcription: Call Whisper API or run whisper.cpp on pro_plus.

Building an ETL pipeline

Extract data from an API, transform each batch in parallel, and load results into Postgres — with per-stage compute sizing and isolated failure recovery.

Extract → fan-out Transform in parallel → Load into Postgres over private network.

The pipeline orchestrator

export const runPipeline = task(
  { name: 'runPipeline', plan: 'starter', timeoutSeconds: 7200 },
  async function runPipeline(apiEndpoint: string) {

    // 1. Extract: fetch all pages from API (serial — needs cursor)
    const batches = await extractRecords(apiEndpoint)

    // 2. Transform: enrich each batch in parallel
    const enriched = await Promise.all(
      batches.map(batch => transformBatch(batch))
    )

    // 3. Load: bulk insert all records into Postgres
    const allRecords = enriched.flat()
    const { inserted } = await loadToPostgres(allRecords)

    return { total: inserted }
  }
)

Extract task — paginated API fetch

export const extractRecords = task(
  { name: 'extractRecords', plan: 'starter', timeoutSeconds: 1800,
    retry: { maxRetries: 3, waitDurationMs: 2000 } },
  async function extractRecords(apiEndpoint: string, batchSize = 100) {
    const batches: Record[][] = []
    let cursor: string | undefined
    do {
      const url = `${apiEndpoint}?limit=${batchSize}${cursor ? `&cursor=${cursor}` : ''}`
      const res = await fetch(url)
      if (!res.ok) throw new Error(`API ${res.status}`)
      const data = await res.json()
      batches.push(data.records)
      cursor = data.nextCursor
    } while (cursor)
    return batches
  }
)

Load task — bulk insert over private network

import { Pool } from 'pg'
const pool = new Pool({ connectionString: process.env.DATABASE_URL })

export const loadToPostgres = task(
  { name: 'loadToPostgres', plan: 'standard', timeoutSeconds: 900,
    retry: { maxRetries: 3 } },
  async function loadToPostgres(records: EnrichedRecord[]) {
    const client = await pool.connect()
    try {
      await client.query('BEGIN')
      for (const r of records) {
        await client.query(
          `INSERT INTO leads (email, region, score) VALUES ($1, $2, $3)
           ON CONFLICT (email) DO UPDATE SET score = $3`,
          [r.email, r.region, r.score]
        )
      }
      await client.query('COMMIT')
      return { inserted: records.length }
    } catch (e) { await client.query('ROLLBACK'); throw e }
    finally { client.release() }
  }
)
Scheduling: Run this pipeline on a schedule by creating a Render cron job that calls render.workflows.startTask('my-workflow/runPipeline', [...]).

AI agent with tool calling

Build an agent that reasons with an LLM, dispatches tool calls as chained tasks, and loops until it reaches a final answer — with each tool running in its own isolated instance.

Why Workflows for AI agents

Tool calls fail independently. A web search might time out while the LLM call succeeds. Without task-level retries, you'd restart the entire agent loop for one flaky tool.

Tools need different resources. An LLM orchestrator needs minimal compute. A code execution tool might need 4 GB of RAM. Workflows lets you right-size each tool.

Agent loops can run for minutes. 5–10 tool calls with LLM reasoning between each is too long for a serverless function, too bursty for an always-on worker.

Agent loop calls LLM, dispatches tool tasks, accumulates context, repeats until answer.

Tool tasks

import { task } from '@renderinc/sdk/workflows'

export const webSearch = task(
  { name: 'webSearch', plan: 'starter', timeoutSeconds: 60,
    retry: { maxRetries: 3, waitDurationMs: 1000 } },
  async function webSearch(query: string): Promise<string> {
    const res = await fetch(`https://api.search.com/v1?q=${encodeURIComponent(query)}`)
    if (!res.ok) throw new Error(`Search API ${res.status}`)
    return (await res.json()).results.map((r: any) => r.snippet).join('\n')
  }
)

export const runCode = task(
  { name: 'runCode', plan: 'pro', timeoutSeconds: 120,  // 2 CPU for compute
    retry: { maxRetries: 1 } },
  async function runCode(code: string, language: string): Promise<string> {
    const { stdout } = execSync(`echo '${code}' | ${language}`, { timeout: 30000, encoding: 'utf-8' })
    return stdout
  }
)

Agent loop

import { webSearch, queryDatabase, runCode } from './tools'
const TOOLS = { webSearch, queryDatabase, runCode } as const

export const agentLoop = task(
  { name: 'agentLoop', plan: 'standard', timeoutSeconds: 600,
    retry: { maxRetries: 1 } },
  async function agentLoop(userPrompt: string, maxIterations = 8) {
    const messages: Message[] = [{ role: 'user', content: userPrompt }]
    let toolCalls = 0

    for (let i = 0; i < maxIterations; i++) {
      const llmResponse = await callLLM({ model: 'gpt-4o', messages, tools: [...] })

      if (llmResponse.type === 'answer')
        return { answer: llmResponse.content, toolCalls }

      // Dispatch tool — runs in a SEPARATE instance with its own retry
      const { toolName, args } = llmResponse.toolCall
      const toolResult = await TOOLS[toolName](...Object.values(args))
      toolCalls++

      messages.push(
        { role: 'assistant', content: `Calling ${toolName}` },
        { role: 'tool', content: toolResult }
      )
    }
    return { answer: 'Max iterations reached', toolCalls }
  }
)
Key design: Each await TOOLS[toolName](...) triggers a chained run in a separate instance. The agent's conversation state stays in the parent. If webSearch fails and retries, the parent's messages array is untouched.

Workflow → Orchestrator task

Temporal
// workflows.ts
import { proxyActivities }
  from '@temporalio/workflow'

const { fetchUser, enrichData,
  sendEmail } = proxyActivities({
    startToCloseTimeout: '5m',
    retry: {
      maximumAttempts: 3,
      backoffCoefficient: 2,
    },
  })

export async function
  onboardUser(userId: string) {
  const user = await fetchUser(userId)
  const data = await enrichData(user)
  await sendEmail(user.email,
    'Welcome!', data.html)
  return { ok: true }
}
Render Workflows
// orchestrator.ts
import { fetchUser, enrichData,
  sendEmail } from './tasks'
// No proxy needed!

export const onboardUser = task(
  { name: 'onboardUser',
    plan: 'starter',
    timeoutSeconds: 600 },
  async function
    onboardUser(userId: string) {
    const user = await
      fetchUser(userId)
    const data = await
      enrichData(user)
    await sendEmail(user.email,
      'Welcome!', data.html)
    return { ok: true }
  }
)

What you gain

No server cluster. No Temporal Server or Cloud subscription. No worker fleet. No long-running processes to manage. No determinism constraints. Tasks are regular functions — call APIs, use Date.now(), anything goes. No proxy boilerplate. Import and call task functions directly. Per-task compute sizing. Each task gets its own instance type.
